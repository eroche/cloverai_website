# Clover.AI Website Build Memo
## Comprehensive brief for the first public website refresh

## 1. Purpose of this memo

This memo is the single source of truth for the first refreshed Clover.AI website.

The builder will have access to:
- this memo
- the current live Clover.AI site, only as a source of reusable visual assets such as logo, colors, icons, backgrounds, or layout ideas

The builder should **not** rely on the current live site for content strategy, page architecture, messaging hierarchy, or service positioning. The current site is outdated, broad, and was effectively a placeholder. The new site must be based on the strategy clarified in recent work: Clover should be positioned as an ultra-fast, trainable, CPU-based NLP infrastructure company for LLM-era systems, with anonymization as the first commercial wedge and the broader CLOVER platform story underneath.

---

## 2. What the website is trying to do

This website is **not** trying to fully explain the whole Clover vision, the entire research program, or every possible product application.

The website has only three jobs:

1. explain what Clover.AI is in a sharp, modern, credible way  
2. make the first commercial wedge legible  
3. generate qualified inbound contact through a form

The site should help the right visitor quickly understand:

- Clover is not a chatbot company
- Clover is not a generic AI consultancy
- Clover is not merely an LLM wrapper
- Clover is a serious language infrastructure company built around the CLOVER programming language
- Clover’s first practical offer is custom anonymization/redaction for enterprise text flows
- Clover’s larger platform supports NER, extraction, tokenization, routing, and related language-processing tasks

This is directly aligned with the current strategy and with the repo context, which both emphasize that Clover is best framed as a compiled language/runtime platform and a hybrid symbolic + LLM infrastructure layer, while warning against overly broad service-oriented positioning.

---

## 3. Core strategic positioning to reflect on the site

The site must express the following strategic idea clearly:

**Clover.AI is an ultra-fast, trainable, CPU-based NLP infrastructure company for LLM-era systems.**

A second supporting idea must also be clear:

**LLMs help create or train the model; CLOVER runs the model in production.**

A third supporting idea must be visible early:

**The first commercial wedge is deterministic anonymization/redaction for sensitive enterprise text flows.**

The website should present these in exactly that order of abstraction:
- broad platform statement first
- concrete commercial wedge second
- platform expansion story third

This structure matters. The strategy work concluded that Clover should lead with the broader platform frame in the hero, but present anonymization as the first concrete offering immediately below that.

---

## 4. Site scope

The first version should be a **small site**, not a large brochure site.

### Required pages
1. **Homepage**
2. **Technology page** (or “CLOVER Technology”)
3. **Contact page** or homepage-integrated contact section, but there must be a proper form

### Not required for v1
Do not create separate pages for:
- Services
- About Us
- Team
- Industries
- Case Studies
- Blog
- Careers
- Research
- Patents
- Partners

Those can all come later if needed. For now they increase breadth and weaken the wedge.

---

## 5. High-level site architecture

### Recommended top navigation
- Home
- Technology
- Contact

Nothing else unless there is a strong implementation reason.

### Footer
Minimal footer:
- Clover.AI wordmark/logo
- one-line company description
- navigation links
- contact link
- legal/privacy placeholders if needed

---

## 6. Tone and visual direction

### Tone
The desired tone is:

- sparse
- technical
- serious
- modern
- restrained
- developer-grade
- enterprise-capable
- intellectually strong without sounding academic

The site should feel closer to:
- technical infrastructure
- advanced enterprise tooling
- serious systems software

and farther from:
- agency marketing
- “AI transformation” consultancy language
- glossy generic startup copy
- academic lab pages
- consumer SaaS friendliness

### Design direction
The site should visually lean toward:
- clean layouts
- generous whitespace
- controlled typography
- dark or neutral technical palette if consistent with current brand
- structured sections
- sharp information hierarchy
- minimal decorative clutter

It should not feel:
- playful
- crowded
- highly animated
- illustration-heavy
- template-like
- packed with cards everywhere

### Reuse from current live site
The builder may reuse from the current site:
- logo
- color palette
- background textures
- subtle graphic motifs
- icons, if they are clean and non-generic
- any visual asset that reinforces technical seriousness

The builder should **not** reuse from the current site:
- page structure
- section order
- service taxonomy
- placeholder-style messaging
- inflated capability lists
- team/about content blocks unless explicitly reworked
- contact wording if it looks like stock placeholder copy

---

## 7. Claims policy

This part is extremely important.

### Allowed hard quantitative claim
At this stage, the **only** hard quantitative performance claim allowed on the website is:

**500,000 words per second on a single CPU**

This may appear if presented cleanly and confidently.

### Avoid for now
Do **not** place these on the site unless separately approved:
- number of supported languages such as 30+ or 40+
- patent counts
- exact benchmark comparisons against named systems
- precision/recall figures
- CPU footprint numbers
- regulatory/compliance claims like HIPAA-compliant unless legally validated
- named enterprise relationships unless explicitly approved
- claims that sound like audited external validation if they are not

This matches the strategic guidance that Clover’s claims should stay disciplined and that the site must not outrun proof or become broad and boastful.

---

## 8. Messaging hierarchy

The builder should preserve this exact hierarchy.

### Message layer 1
**Clover.AI builds ultra-fast, trainable, CPU-based NLP infrastructure for LLM-era systems.**

### Message layer 2
**Clover uses LLMs to help create language models, then runs those models as deterministic CLOVER programs on CPU.**

### Message layer 3
**The first commercial wedge is custom anonymization/redaction for enterprise text flows.**

### Message layer 4
**The same underlying platform extends to NER, extraction, tokenization, routing, and structured language processing.**

### Message layer 5
**CLOVER is a real programming language for natural language.**

### Message layer 6
A light credibility layer:
- deep industrial NLP background
- long-built foundational technology
- not trend-following

The site should not lead with the long-range theory/research layer. That should remain background credibility, not homepage foreground.

---

## 9. What the site must not become

The website must not read like:

- a generic AI agency
- a consulting brochure
- an LLM wrapper
- a symbolic-NLP nostalgia page
- a laundry list of services
- a founder résumé page
- a research institute homepage

The strategic risk to avoid is breadth without focus. The older materials already show too much breadth and too much service-style positioning. The new site must correct that.

---

## 10. Homepage: required structure

The homepage should do almost all the strategic work.

### Section 1. Hero

#### Purpose
Introduce Clover at the correct level: broad platform first, not anonymization first.

#### Content goals
The visitor should understand in 5 seconds:
- this is infrastructure, not a chatbot
- it is technical and serious
- it is for the LLM era
- it has a deterministic CPU execution story

#### Recommended hero headline
**Ultra-fast, trainable, CPU-based NLP infrastructure for LLM-era systems**

#### Recommended hero subhead
**Clover.AI uses LLMs to help create language models, then deploys them as deterministic CLOVER programs running at high speed on CPU.**

Alternative shorter version:
**Build language intelligence with LLMs. Run it in production with CLOVER.**

#### Supporting proof line
Optional short line:
**Up to 500,000 words per second on a single CPU.**

#### Primary CTA
**Request a demo**

#### Secondary CTA
**View technology**

#### Layout guidance
- left-aligned text preferred
- one visual motif or background pattern is enough
- avoid over-illustration
- do not place too many feature bullets in the hero
- do not insert generic badges, fake social proof, or partner logos unless approved

---

### Section 2. Why now / problem framing

#### Purpose
Explain why runtime LLM-only systems create demand for Clover.

#### Content themes
- LLMs are powerful but expensive to run continuously
- runtime inference is not ideal for every text-processing step
- privacy and control matter
- deterministic always-on language processing is a missing layer

#### Suggested heading
**LLMs are powerful. Runtime language processing is still too expensive, too opaque, and too hard to control.**

#### Suggested body copy
“Many production AI systems use LLMs for tasks that do not need probabilistic runtime inference on every text flow. Clover adds a deterministic language-processing layer for redaction, extraction, routing, and structure around LLM systems.”

#### Layout guidance
Keep this section concise. One paragraph plus 3 short bullets is enough.

Possible bullets:
- deterministic behavior
- low runtime cost
- deployable on CPU

---

### Section 3. First commercial wedge: anonymization

#### Purpose
This is the first concrete commercial section. It should make the site feel real, practical, and sellable.

#### Key message
Clover’s first offer is custom anonymization/redaction for enterprise text flows.

#### Suggested heading
**Start with deterministic anonymization for sensitive text flows**

#### Suggested body copy
“Clover’s first commercial offering is custom anonymization and redaction for enterprise text pipelines. Sensitive information can be detected and removed before text reaches LLMs, agents, search systems, or downstream applications.”

#### Supporting bullets
- redact sensitive text before LLM processing
- reduce leakage risk in agent and document workflows
- deploy a custom solution for your environment
- deterministic behavior across runs

#### Optional mini-CTA
**Discuss your use case**

#### Layout guidance
This section should feel like the first real product section, not a long feature list.

---

### Section 4. Platform expansion: one infrastructure layer, multiple capabilities

#### Purpose
Show that anonymization is the entry point, not the whole company.

#### Key message
The same underlying CLOVER platform extends to related NLP capabilities.

#### Suggested heading
**One language infrastructure layer, multiple production capabilities**

#### Suggested short intro
“Clover is not a single-point product. The same infrastructure layer extends from redaction into broader language-processing tasks.”

#### Capability list
Use a concise card/grid set:
- Anonymization / Redaction
- Named Entity Recognition
- Information Extraction
- Tokenization
- Routing / Categorization

Do not add too many more.

#### Important wording rule
Do not present these as separate consulting services. Present them as capabilities of one platform.

---

### Section 5. Why Clover is different

#### Purpose
This is the core differentiation block.

#### Suggested heading
**Why Clover is different**

#### Use 4 proof cards only
1. **Deterministic**  
   What runs in production behaves consistently.

2. **CPU-fast**  
   Designed for very high-speed execution on CPU.

3. **Trainable**  
   LLMs help create the model; CLOVER runs it.

4. **Programmable**  
   Built on a real programming language for natural language.

#### Optional short line beneath cards
“Clover combines symbolic structure with modern model-driven training, without making runtime systems dependent on LLM inference for every text operation.”

---

### Section 6. CLOVER language preview

#### Purpose
Introduce CLOVER as a strategic moat without turning the homepage into a programming-language explainer.

#### Suggested heading
**Built on the CLOVER programming language**

#### Suggested copy
“CLOVER is a programming language for natural language. It provides a compiled runtime layer for redaction, extraction, tokenization, routing, and other structured text-processing tasks.”

#### CTA
**Explore the technology**

#### Layout guidance
This section should be shorter than the dedicated Technology page. Do not go deep into compiler internals, theory, or formal language foundations on the homepage.

---

### Section 7. Credibility block

#### Purpose
Use founder credibility without making the site feel like a one-person consultancy.

#### Important rule
Do **not** create a big “Founder” section with a portrait, long bio, and full résumé.

Instead use a restrained credibility strip or small block.

#### Suggested heading
**Built by deep industrial NLP experience**

#### Suggested copy
“Clover.AI is built by Emmanuel Roche, former co-founder and CTO of Teragram and former Director of Language Processing at SAS, bringing decades of industrial NLP experience to a new generation of trainable language infrastructure.”

This is enough. No longer biography is needed on the homepage.

#### Design guidance
This block should be visually smaller than the product and platform sections.

---

### Section 8. Light multilingual mention

#### Purpose
Acknowledge multilingual capability without making it a main story.

#### Suggested placement
Either:
- at the end of the platform section
- or as one short line near the credibility / technology section

#### Suggested wording
“Designed with multilingual language processing in mind.”

or

“Built for language-aware processing, with multilingual expansion as part of the platform direction.”

#### Important rule
Do not use exact counts like 30+ or 40+ on v1.

---

### Section 9. Final CTA section

#### Purpose
Convert interest into contact.

#### Suggested heading
**Tell us what text flow you need to control**

#### Suggested body copy
“Evaluating privacy-safe AI pipelines, redaction, or structured language processing? Send us a short note and, if helpful, a sample use case.”

#### CTA
Embedded contact form

## 11. Technology page: required structure

This page exists for technical visitors, serious prospects, and anyone who wants to understand what CLOVER is without cluttering the homepage.

### Technology page objective
Explain the platform more deeply, but still commercially.

Do not make it a research paper.  
Do not make it a full compiler manual.

### Technology page sections

#### 1. Intro hero
Suggested heading:
**The CLOVER technology stack**

Subhead:
“A compiled language and runtime for deterministic, trainable natural language processing.”

#### 2. What CLOVER is
Explain:
- programming language for natural language
- compiled execution model
- infrastructure layer around LLMs
- suitable for redaction, extraction, tokenization, routing, and structure

#### 3. How CLOVER works with LLMs
Key message:
- LLMs help train or create the model
- CLOVER executes it in production
- runtime systems do not need LLM inference for every text step

#### 4. Deployment roles
Use 4–5 compact blocks:
- Preprocessing
- Redaction
- Extraction
- Routing
- Tokenization

#### 5. Performance / execution
This is the only page where the performance claim can appear slightly more prominently.

Allowed phrasing:
**Designed for very high-speed CPU execution, including internal benchmarks around 500,000 words per second on a single CPU.**

If the owner prefers stricter phrasing, the builder should use:
**500,000 words per second on a single CPU**

#### 6. Broader platform direction
Mention:
- anonymization
- NER
- extraction
- tokenization
- structured language processing

#### 7. CTA
**Request a demo**  
or  
**Discuss your use case**

- CPU-based

- text flows
- structured language processing
- redaction
- extraction

- control

- transform your business
- next-generation AI solutions
- end-to-end AI transformation
- human-like understanding
- magical
- seamless
- cutting-edge everywhere
- full-stack AI
- enterprise-grade if unsupported

- crisp
- declarative

- low adjective density
- little hype
- minimal jargon stacking

## 18. Acceptance criteria

The first version is successful if, after landing on the homepage, a serious visitor can correctly say:

- Clover is an NLP infrastructure company
- it works with LLMs but is not just an LLM wrapper
- it uses CLOVER as a programming language/runtime
- its first productized commercial wedge is anonymization
- it can extend to other language-processing capabilities
- there is a clear way to contact the company

The first version is unsuccessful if the visitor leaves thinking:

- this is a generic AI consulting firm
- this is a broad menu of services
- this is a founder-personal site
- this is a research lab homepage
- this is a chatbot company
- this is symbolic NLP nostalgia without a present commercial wedge

---

## 19. Final instruction to the builder

Use the old site only as a visual asset library.  
Use this memo as the strategy, structure, copy, and design authority.

The new site must feel narrower, sharper, and more serious than the current one. It must present Clover as a platform company with a concrete first wedge, not as a placeholder brochure.

This brief follows the current strategic direction: Clover as a compiled, trainable, CPU-based NLP infrastructure layer around LLM-era systems; anonymization as the first commercial wedge; broader platform capability beneath that; and founder credibility used in a restrained way.
